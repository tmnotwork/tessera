# データ（正本 DB）の構成

このフォルダ（`assets/data/`）は **英語学習アプリ用の正本 DB** です。参考書・アプリはここから変換して作る。  
**Cursor で編集する対象はこの assets/data/ 一式のみ。** 形式は以下に従うこと。

## 方針

- **DB と参考書は分離**：ここでは「知識・問題・試験マッピング」だけを管理する。参考書の形にはしない。
- **あとでいかようにでも使える**：形式は汎用にし、必要なときに参考書用などへ変換する。
- **単一の正本**：知識・問題・試験の対応は、この assets/data/ 以外に持たない。
- **この assets/data/ の位置づけ**：いまの assets/data/ は「最後の参考書の元データ」。データサイエンスで「何を入れるか」を決めるための「事実を記録する DB」（例：この単語はある大学の受験で何度出現した、など）は別層として検討する。詳細は `../../docs/DB_ヒアリング結果.txt` の【データサイエンス前提と「事実を記録するDB」の整理】を参照。

## データ同士の関係（多対多）とつなぎ方

- **知識 ↔ 問題**：多対多。**つなぎ方**＝`questions.json` の各問題に `knowledge_ids[]` を持つ。1 問が複数知識を問う場合は配列に複数 ID を書く。リンク用の別ファイルは作らない。
- **知識 ↔ 試験**：多対多。**つなぎ方**＝`exam_mappings.json` で「ある知識がある試験で必要」の対応を 1 件ずつ管理する（リンク専用）。
- **問題 ↔ 試験**：対応用のデータは持たない。参考書などで「この試験向けの問題」が欲しいときは、「試験 → exam_mappings で知識一覧 → その知識を問う問題を questions から抽出」で求める。

（根拠：正本を二重にしない・Cursor で 1 問追加が 1 ファイルで済む・参考書は「試験→知識→問題」の流れで作る、という要件に合わせた。）

## ナレッジ（知識）の種類と形

- 知識は **「文法・構文」** と **「語彙」** の二つを区別して保存する。
- **採用形**：`knowledge.json` を 1 つとし、各レコードに `type` を持たせる。値は `"grammar"`（文法・構文）または `"vocabulary"`（語彙）。
- **必須**：一つの question に複数の知識が紐づく形にする。`questions.json` の各問題の `knowledge_ids[]` に複数 ID を並べる（文法・語彙の混在可）。この形で「1 問に複数知識が詰め込まれる」ことを満たす。
- **初期の進め方**：最終的には統合されていくとしても、初期は文法と語彙を分けて進める。まず**文法の DB** をしっかり作り込む（特に基礎的な項目）。必要な文法項目を列挙し、それを覚えるための問題＝例文を作っていくことにフォーカスする。
- **最初のパイロット**：1 テーマ（例：仮定法）で実際に DB を作ってみる。その後、徐々に範囲を広げる（仮定法 → 時制、文型、など）。詳細は `../../docs/プロジェクト全体_ブレスト・方向性.txt` の 7. ～ 8. を参照。

## ファイル構成（想定）

| ファイル | 内容 |
|----------|------|
| `knowledge.json` | 知識を最小単位で保存。各レコードの中核は **「タイトル」** と **「その解説」**。`id` と `type`（"grammar" \| "vocabulary"）を必ず持つ。`construction`（bool, 任意）：Construction（文の形・パターン）について説明したカードは `true`。文法には「覚える順番」用の情報を持たせ、参考書で基礎→応用の順に並べる。語彙の順番は「どの試験で必要か」のデータで対応する。詳細は `docs/DB_ヒアリング結果.txt` を参照。 |
| `questions.json` | 一問一答（暗記カード形式）。各レコードに `title`, `question`, `answer`, `explanation`, `supplement`, `question_lang`, `answer_lang`, `knowledge_ids[]` を持つ。言語フィールドは後で読み上げ（TTS）に利用する。 |
| `exam_mappings.json` | 「どの知識がどの試験で必要か」の対応（1 件 = 1 知識–1 試験）。 |

## questions.json のスキーマ

| 項目 | 型 | 必須 | 説明 |
|------|-----|------|------|
| `id` | string | ○ | 一意の ID（例：q001） |
| `title` | string | ○ | タイトル |
| `question` | string | ○ | 問題文 |
| `answer` | string | ○ | 正答 |
| `explanation` | string | △ | 解説（任意） |
| `supplement` | string | △ | 補足（任意） |
| `question_lang` | string | ○ | 問題の言語（ISO 639-1。例：ja, en）。読み上げ用 |
| `answer_lang` | string | ○ | 回答の言語（ISO 639-1）。読み上げ用 |
| `knowledge_ids` | string[] | △ | この問題が問う知識の ID 配列（任意、複数可） |
| `order` | number | △ | 表示順（任意） |

例：
```json
{
  "id": "q001",
  "title": "仮定法過去の英作文",
  "question": "もっと時間があれば、もっと本を読むのになあ。",
  "answer": "If I had more time, I would read more books.",
  "explanation": "仮定法過去の形：If S 過去形, S would 原形",
  "supplement": "",
  "question_lang": "ja",
  "answer_lang": "en",
  "knowledge_ids": ["k004"],
  "order": 1
}
```

## スキーマ・編集ルール

- 各 JSON の項目名・型・必須は、この README または `../../docs/schema.md` で定義する。
- 編集時は **既存の形式を崩さない**。追加するときは既存レコードの形に合わせる。
- ID の付け方や必須項目は、スキーマが決まり次第ここに追記する。

## Cursor で編集するとき

1. **必ず最初に** `.cursor/rules/knowledge-editing.md` を読む
2. 変更するのは `assets/data/*.json` のみ。正本以外に同じ内容を書かない。
3. スキーマ（README または schema.md）を確認してから編集する。
4. 編集後、プロジェクトに検証スクリプトがあれば実行して形式を確認する。

### explanation の書き方（重要）

- **文体は「です・ます」で統一**：解説本文はすべてですます調で書く
- **最初に核心を一文で**：カジュアルすぎない文体で本質を伝える
- **例文は markdown 引用（`>`）で記述**
- **英語と日本語は別行**：英文の直後に改行、日本語訳。間に空行は入れない

例：
```json
"explanation": "仮定法は、現実ではない妄想を語る表現です。\n\n> If I were a bird, I could fly to you.\n> （もし私が鳥だったら、あなたのところへ飛んでいけるのに）"
```
