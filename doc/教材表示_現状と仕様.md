# 教材表示の現状と仕様

## 根本原因：RLS 変更で「認証なしでは見えない」設計になった

**ご指摘のとおり、場当たり的な修正になっていました。** 原因を整理します。

### 何が変わったか（マイグレーション 00012）

- **00002 の頃**: `subjects` / `knowledge` / `questions` などは **anon（未ログイン）にも全開放**（`USING (true)`）だった。  
  → 誰でも同じデータを「共通」で見られた。
- **00012（profiles・ロール導入）**: 既存の **anon 用ポリシーを削除**し、**authenticated（ログイン済み）のみ**に変更した。
  - **subjects**:  
    - 教師（`get_my_role() = 'teacher'`）→ 全操作可  
    - その他の認証ユーザー → **SELECT のみ可**（`subjects: learner read` の `USING (true)`）
  - **knowledge**: 教師のみ全操作可（学習者は別ポリシーで読む想定）

つまり **「ユーザーIDを導入した」というより、「認証なし（anon）でのアクセスをやめた」** ことで、  
**未ログイン or セッションが乗っていないリクエストでは 0 件** になります。

### subjects に user_id はない

- `subjects` テーブルには **user_id カラムはありません**（`name`, `display_order` などだけ）。  
- データは「誰のもの」ではなく **共通マスタ**のまま。  
- 見えなくなる理由は **「そのリクエストが anon になっている」** ことです（認証が効いていないタイミングで読んでいる）。

### 起きがちなパターン

1. **起動直後・セッション復元前**に subjects を読む → anon 扱い → 0 件  
2. **モバイル/デスクトップ**: 未ログインのまま Sync が走る → RLS で subjects が取れずローカルが空 → ログイン後も同期し直さないと空のまま  
3. **教師タブ**: ログイン済みでも、`profiles` に 1 行ないと `get_my_role()` が NULL → 「教師用」の INSERT/UPDATE/DELETE は使えない（SELECT は認証さえあれば可能なはず）

### やるべきだったこと（設計として）

- **未ログインで教材・知識の画面を見せない**。アプリとして「ログインしてから使う」前提にし、未ログイン時は **ログインゲートのみ表示**（学習者でログイン / 教師でログインを選ばせ、ログイン後に 3 タブを表示）。
- **「認証が確定するまで教材系のテーブルを読まない」** ことをはっきり決める。  
- ログイン後の 1 回目取得タイミングを **「認証変更のあと」** に統一する。

以下は、上記を前提にした「現状の実装」です。

## 本質的・完全な状態か（実装の整理後）

### 本質的にしていること

1. **未ログインで教材・知識の画面を見せない**
   - 未ログイン時は **ログインゲートのみ表示**（「ログインしてください」＋学習者でログイン / 教師でログイン）。**3 タブ（学習・知識DB・教師用管理）はログイン後のみ表示**。知識DBタブが未ログインで見えることはない。

2. **認証が確定するまで教材を読まない**
   - **教師用管理**: ログイン済みかつ `get_my_role() = 'teacher'` のときだけ `TeacherAdminPage` を表示し、その時点で初回取得。
   - **知識DBタブ**: 表示されるのはログイン後のみなので、マウント時＝認証済み。initState で 1 回取得。モバイル/デスクトップは同期完了待ちで 2 秒後に 1 回だけ遅延再取得（フォールバック）。

### 残しているフォールバック（意図的に許容）

- **教師用管理の「データを再取得」**: 権限キャッシュ・ロール再取得＋科目再取得。別端末・ロール遅延時の手動リカバリ用。
- **知識DBの「ログイン直後 2 秒遅延 1 回」**: モバイル/デスクトップで Sync 完了前に初回取得するとローカルが空のため、認証確定後に 1 回だけ遅延再取得。毎回 3 秒遅延していたのをやめ、**認証起因の 1 回** に限定。

### 完全かどうか

- **設計方針としては完全**: 「認証確定後にだけ取得」「未ログイン時は取得しない・案内表示」を実装済み。場当たりの「常時 3 秒遅延」「未ログインでも取得して 0 件表示」はやめた。
- **運用上のフォールバックは残している**: 教師の「データを再取得」、知識DBの「ログイン直後 2 秒 1 回」は、ネットワーク・同期タイミングのばらつきに対する保険。必要に応じて将来「同期完了イベントで再取得」に置き換え可能。

## 現状（確認結果）

- **Supabase**: `subjects` テーブルに **1件** 存在する。`profiles` に教師（role=teacher）の行があり、RLS 上は問題ない。
- **教師用管理タブ**（教材管理）: Supabase の `subjects` を **直接** 参照している。ログイン済み・role=teacher ならここでは科目一覧が表示される想定。
- **知識DBタブ**（真ん中のタブ）: 
  - **Web**: Supabase を直接参照。教師・学習者どちらでも科目が見える。
  - **モバイル/デスクトップ**: **ローカルDB（`local_subjects`）** を参照する。データは SyncEngine が Supabase から Pull して入れる。**起動時に未ログインのまま同期が走ると、RLS で subjects が取れずローカルが空のまま**になる。そのあとログインしても、それまで同期が再実行されていなかったため「科目がありません」のままになることがあった。

## 対応済み

- **ログイン後に同期を1回実行する**（`main.dart` の `_onAuthChanged`）。モバイル/デスクトップでログインすると `syncIfOnline()` が呼ばれ、認証付きで Supabase から subjects/knowledge がローカルに取り込まれる。
- **教師用管理**: 表示されるのはログイン済みかつ role=teacher のときのみ。その時点で初回取得。科目が空のときは「データを再取得」（権限キャッシュクリア＋ロール再取得＋科目再取得）をフォールバックとして提供。
- **未ログイン時はログインゲートのみ**: 3 タブは出さず「ログインしてください」と学習者/教師のログイン選択。ログイン後に 3 タブを表示。
- **知識DBタブ**: ログイン後のみ表示。初回取得は **1 フレーム遅延**（`addPostFrameCallback`）でセッション反映後に実行。モバイル/デスクトップは同期完了待ちで **2 秒後に 1 回だけ** 遅延再取得。
- **教師用管理**: 同様に初回取得は 1 フレーム遅延。接続テストで 0 件のときは「科目が0件です。RLS・権限または接続先プロジェクトを確認してください。」と表示。

## ログインしてもDBが見えない問題は本当に解決しているか

### 「本当に」解決しているか：検証を入れた

**ログイン直後に「科目が取れるか」を 1 回だけ検証し、結果を画面に出す**ようにした。

- **流れ**: ログイン → `_refreshRole()`（`fetchRole()` で認証付きリクエスト）→ **同じセッションで `subjects` を 1 回取得** → 成功なら何も出さない、**0 件なら「科目が0件です。接続先・RLSを確認してください。」、例外なら「科目の取得に失敗しました: {例外}」** を画面上部のバナーで表示。
- ユーザーは **ログインした瞬間に「取れた／0件／エラー」のどれか** が分かる。隠れていて「見えない」だけにしない。

### アプリ側の保証

1. **未ログインでは取得しない**  
   3 タブはログイン後のみ表示。

2. **ログイン直後の検証**  
   `_refreshRole()` 内で、認証付きで `subjects` を 1 回取得。0 件 or 例外なら `_postLoginSubjectsCheck` にメッセージを入れ、画面上部に表示（「閉じる」で非表示可能）。

3. **初回取得のタイミング**  
   教師用管理・知識DBの初回取得は 1 フレーム遅延（`addPostFrameCallback`）。モバイル/デスクトップはログイン後に `syncIfOnline()` を 1 回実行し、知識DBは 2 秒後に 1 回だけ再取得。

### まだ残りうる原因（環境・設定）

ログインしても 0 件のままなら、次を確認する。

| 原因 | 確認方法 |
|------|----------|
| **接続先プロジェクトが違う** | アプリの `.env`（または Supabase 設定）の `SUPABASE_URL` / `SUPABASE_ANON_KEY` が、科目データを入れている **同じプロジェクト** か。 |
| **RLS やマイグレーション未適用** | 00012（profiles・RLS 変更）と 00013（`ensure_my_profile`）がそのプロジェクトに適用されているか。Supabase Dashboard → SQL でマイグレーション履歴を確認。 |
| **profiles に自分がいない** | 教師の場合は `ensure_my_profile()` で 1 行できる。ログイン後「データを再取得」や「接続テスト」で 0 件なら、Dashboard の `profiles` に `auth.uid()` の行があるか確認。 |
| **subjects にデータがない** | Dashboard の `subjects` に 1 件以上あるか。 |

教師用管理で **「接続テスト」** を実行し、  
- **エラー** → ネットワーク or RLS/ポリシー  
- **「科目が0件です」** → 接続先・RLS・データのどれか  

と切り分けできる。

## 画面ごとのデータ元

| 画面 | Web | モバイル/デスクトップ |
|------|-----|------------------------|
| 教師用管理（教材管理） | Supabase 直接 | Supabase 直接 |
| 知識DBタブ | Supabase 直接（SubjectRepositorySupabase） | ローカルDB（SubjectRepositoryLocal） |
| 学習タブ（学習者ホーム） | Supabase 直接 | Supabase 直接 |

## まだ教材が見えない場合

1. **教師用管理**で見えない: 画面上部の **「接続テスト」（Wi-Fiアイコン）** を押してエラーが出ないか確認。エラーなら RLS またはネットワーク要因。
2. **知識DBタブ**で見えない（モバイル/デスクトップ）: ログイン直後は同期が完了するまで数秒かかることがある。**再読み込み**ボタンを押す。それでも空なら、オフラインまたは同期エラーの可能性あり。
