# 認証・ログイン 全体設計

本番リリースを見据えた認証フローの全体像と、エラー・入力検証のルールをまとめる。

---

## 前提：Supabase の設定が必須

**確認メール**および**パスワードリセット用メール**のリンクを利用する場合、**Supabase の Site URL / Redirect URLs の設定が必須**です。アプリ側では変更できません。

- **Site URL に `https://supabase.com` は使わない。** あなたのアプリのユーザーを第三者サイトに飛ばすことになり不適切です。必ずあなたが公開・管理している URL を設定する。
- **開発**: (A) Confirm email を OFF にしてメール確認を省略する（設定不要）、または (B) `static/email-confirmed.html` を GitHub Pages / Vercel / Netlify で公開し、その URL を Site URL と Redirect URLs に設定する。
- **本番**: Site URL を本番のルート URL にし、Redirect URLs に同じ URL を追加する。確認完了用に `static/email-confirmed.html` を本番ドメインで公開する。

**具体的な手順は `doc/Supabase_本番設定手順.md` にのみ記載する。デプロイ前・確認メールを有効にする前に必ず実施すること。**

---

## 1. 役割とフロー一覧

| 対象 | ログイン | 新規登録 | 確認メール再送 | パスワードリセット | ログアウト |
|------|----------|----------|----------------|---------------------|------------|
| **教師** | メール+パスワード | あり（メール確認あり） | あり | あり（メールでリセット案内） | 教材管理画面・学習タブの両方から可能 |
| **学習者** | ユーザーID+パスワード | なし（教師が発行） | 該当なし | 教師に依頼（アプリ内では案内のみ可） | 学習ホームのAppBarから |

- 教師と学習者は **別アカウント**（auth.users 上で 1 ユーザー = 1 役割）。
- 教師は `profiles.role = 'teacher'`、学習者は `@tessera.local` メール＋`profiles.role = 'learner'` で区別。

---

## 2. 教師：ログイン

- **入力**: メールアドレス（trim）、パスワード。
- **検証**: メール未入力・パスワード未入力は送信不可。メール形式は簡易チェック（`@` を含む等）推奨。
- **成功時**: 教材管理タブ（TeacherAdminPage）を表示。`fetchRole()` で `teacher` であることを確認。
- **エラー分類と表示**:
  - `email_not_confirmed` / "Email not confirmed"  
    → 「メールアドレスがまだ確認されていません。登録時に送信されたメールのリンクから確認を完了してください。届かない場合は『確認メールを再送』を押してください。」  
    アクション: 「確認メールを再送」
  - `invalid_login_credentials` / "Invalid login credentials"  
    → 「メールアドレスまたはパスワードが正しくありません。」
  - その他（ネットワーク等）  
    → 「ログインに失敗しました。接続を確認するか、しばらく経ってからお試しください。」（本番では詳細を出しすぎない）

---

## 3. 教師：新規登録

- **入力**: メールアドレス（trim）、パスワード。
- **検証**:
  - 未入力チェック。
  - メール形式（簡易）。
  - パスワード長（例: 8 文字以上）を必須とする。
- **既に登録済みのメールで登録した場合**:
  - Supabase は「Confirm email」ON のとき、既存メールでも 200 を返しうる（セキュリティ上の意図）。  
    その場合 `AuthResponse.user.identities` が空なので、**「このメールアドレスはすでに登録されています。ログインタブからログインしてください。」** と表示し、ログインタブへ切り替える。
  - `AuthException` で "already" / "registered" / "exists" 等が含まれる場合も同メッセージ＋ログインタブへ。
- **成功時（新規作成された場合）**:  
  「登録しました。確認メールを送信しました。メールのリンクをクリックして確認を完了してから、ログインしてください。」  
  ログインタブへ切り替える。

---

## 4. 確認メール再送

- **対象**: 教師の未確認メールアドレス。
- **入力**: ログインタブのメール欄（trim）。
- **成功時**: 「確認メールを再送しました。届くまで少々お待ちください。迷惑メールもご確認ください。」
- **エラー**:
  - 既に確認済みなどで Supabase がエラーを返した場合  
    → 「このメールはすでに確認済みです。ログインをお試しください。」などに統一。
  - その他  
    → 「再送に失敗しました。メールアドレスを確認するか、しばらく経ってからお試しください。」

---

## 5. パスワードリセット（教師）

- **フロー**: ログイン画面で「パスワードを忘れた」→ メール入力 → `resetPasswordForEmail(email)` を呼ぶ。
- **成功時**: 「パスワードリセット用のメールを送信しました。メールのリンクから新しいパスワードを設定してください。」
- **エラー**: 登録されていないメールでも Supabase は「送信した」と返す場合がある（列挙対策）。  
  アプリ側では「送信しました」で統一してよい。  
  明示的な API エラーが出た場合のみ「送信に失敗しました。メールアドレスを確認してください。」等を表示。

---

## 6. 学習者：ログイン

- **入力**: ユーザーID（trim）、パスワード。内部で `userId@tessera.local` に変換して signInWithPassword。
- **検証**: 未入力は送信不可。
- **エラー**: ID/パスワード誤り・ネットワーク等は **「IDまたはパスワードが正しくありません」** に統一（列挙対策）。
- 学習者には新規登録・確認メール再送はない（教師がアカウント発行）。

---

## 7. ログアウト

- **教師**:  
  - 教材管理画面（TeacherAdminPage）の AppBar にログアウトを配置。  
  - ロール未取得で「このタブは教師専用です」と出ている画面にもログアウトを配置済み。
- **学習者**:  
  - 学習ホーム（LearnerHomeScreen）の AppBar にログアウトを配置。
- **共通**: `appAuthNotifier.logout()`（Supabase signOut）を呼び、認証状態がクリアされればログイン画面に戻る。

---

## 8. 学習者管理（教師向け）

- **一覧**: `profiles` の `role = 'learner'` を取得（教師は RLS で参照可）。教材管理タブ内「学習者管理」で表示。
- **追加**: ユーザーID・パスワード・表示名（任意）を入力。Supabase Auth Admin API で `userId@tessera.local` のユーザーを作成し、`profiles` を `role=learner`, `user_id=userId` に更新。**要・SUPABASE_SERVICE_ROLE_KEY**（.env に設定）。未設定時は一覧のみ表示可能。
- **削除**: Auth Admin API で該当ユーザーを削除。`profiles` は CASCADE で削除される。要・サービスロールキー。
- **パスワードリセット**: Auth Admin API で該当ユーザーのパスワードを更新。要・サービスロールキー。

## 10. 本番運用メモ

- **Confirm email**: 本番では ON 推奨。開発時は OFF にして即ログイン可能にできる（`doc/教師ログイン_メール確認.md` 参照）。
- **Site URL / Redirect URLs**: 上記「前提」のとおり **Supabase の設定が必須**。入力値・手順は `doc/Supabase_本番設定手順.md` に記載。未設定のままでは確認・リセットリンクは動作しない。
- **エラーメッセージ**: 本番では「このメールは登録されていません」などと返さず、ログイン/登録の失敗は同じ文言にまとめる方針でよい（列挙対策）。
- **学習者管理**: 追加・削除・パスワードリセットを使う場合は、.env に `SUPABASE_SERVICE_ROLE_KEY` を設定する。サービスロールキーは厳重に管理し、クライアント配布物に含めないよう注意（Web では未使用。モバイル/デスクトップで .env から読み込む）。
