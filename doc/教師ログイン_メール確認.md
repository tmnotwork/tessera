# 教師ログイン・確認メール（運用手順）

> 認証の全体像は [認証・ログイン_全体設計.md](認証・ログイン_全体設計.md) を参照。

---

## 確認メールのリンクを動かすには Supabase の設定が必須

確認メール・パスワードリセットのリンクは **Supabase の Site URL** で飛び先が決まります。**設定しないとリンクは動作しません。**

**やること**: `doc/Supabase_本番設定手順.md` の手順どおりに行う。

- **Site URL に `https://supabase.com` は使わない。** あなたが公開している URL を設定する。
- 今すぐログインだけできればよい場合: Supabase で **Confirm email** を OFF にすると、メール確認なしでログインできる（Site URL の設定不要）。
- 確認メールのリンクも動かしたい場合: `static/email-confirmed.html` を GitHub Pages / Vercel / Netlify で公開し、その URL を Site URL と Redirect URLs に設定する。
- 本番リリース時: Site URL = 本番のルート URL、Redirect URLs に同じ URL を追加。`static/email-confirmed.html` を本番で公開する。

設定後は、**「確認メールを再送」を押して、新しく届いたメールのリンクから開くこと。** 設定前に届いたメールのリンクは使わない。

---

## メールが届かない場合

- 迷惑メールフォルダを確認する。
- 数分待ってから再度試す。
- アプリのログイン画面で「確認メールを再送」を利用する。
- 開発中は、Supabase の **Authentication** → **Providers** → **Email** で **Confirm email** を OFF にすると、メール確認なしでログインできる。

## 既存ユーザーに教師権限が付与される仕組み（根本）

- **教師権限の有無**は、Supabase の **`profiles` テーブルにそのユーザーの行があり、`role='teacher'` かどうか**で決まる。
- **新規登録**（メールで signUp）: `auth.users` に INSERT されると DB トリガー `handle_new_user()` が動き、`profiles` に `role=teacher` で1行が自動作成される。**アプリの「再読み込み」は関係ない。**
- **既存ユーザー**（トリガー適用前に登録したアカウントなど）: `profiles` に行が無い状態でログインし、アプリがロールを取得するとき（`fetchRole()`）、**その処理のなかで** Supabase の RPC **`ensure_my_profile()`** が呼ばれる。この RPC が `profiles` に1行挿入し、メールが `@tessera.local` でなければ `role=teacher`、そうでなければ `role=learner` になる。  
  **つまり「権限を付与している」のはこの RPC の実行結果であり、「再読み込み」は「ロール取得（fetchRole）を再度実行する」だけ。** 再読み込みで RPC がもう一度呼ばれ、そのときに初めて profile が作られる場合がある、という動きになる。
- **根本的に必要なこと**: 既存ユーザーにも教師権限を付けるには、**マイグレーション 00013 を Supabase に適用し、`ensure_my_profile()` が存在・成功する状態にしておくこと。** 未適用だと RPC が無く、何度再読み込みしても profile は作られない。

---

## 00013 を適用する（ensure_my_profile を使えるようにする）

既存ユーザーに教師権限を付与するには、**次の手順で 00013 を必ず適用する。**

1. **Supabase ダッシュボード** を開く（https://supabase.com/dashboard）→ 対象プロジェクトを選択する。
2. 左メニュー **SQL Editor** を開く。
3. **New query** で新規クエリを開く。
4. リポジトリの **`doc/apply_00013_ensure_profile.sql`** を開き、**ファイルの中身をすべてコピー**する。
5. SQL Editor の編集欄に **貼り付けて Run**（または Execute）する。
6. エラーが出ずに完了すれば、`ensure_my_profile()` が利用可能になっている。既存ユーザーはアプリで教材管理タブを開くか「再読み込み」すると、profile が自動作成され、教師として入れます。

（Supabase CLI で `supabase db push` を使っている場合は、`supabase/migrations/00013_ensure_profile_for_existing_user.sql` が未適用なら push で適用される。）

## ログインはできるが「教材管理」に入れない場合

1. **上記「00013 を適用する」をまだ実行していなければ実行する。** これが根本対応。
2. 適用済みなら、教材管理タブで **「再読み込み」** を押す（ロール取得が再度走り、その過程で `ensure_my_profile()` が呼ばれ、未作成なら profile が作られる）。
3. それでも付かない場合: 下記「既存ユーザー分の profile を一括で作る」を Supabase の SQL Editor で実行する。
4. 学習者アカウント（`@tessera.local`）でログインしている場合は、教師用メールでログインし直す。

### 既存ユーザー分の profile を一括で作る（運用で1回だけ）

`auth.users` にいるが `profiles` にまだ行が無いユーザーに、教師／学習者を振り分けて `profiles` を1行ずつ作る。**Supabase ダッシュボード → SQL Editor で実行する（要: プロジェクトの実行権限）。**

```sql
INSERT INTO public.profiles (id, role, display_name)
SELECT u.id,
       CASE WHEN u.email LIKE '%@tessera.local' THEN 'learner' ELSE 'teacher' END,
       u.email
FROM auth.users u
LEFT JOIN public.profiles p ON p.id = u.id
WHERE p.id IS NULL
ON CONFLICT (id) DO NOTHING;
```

実行後、該当ユーザーはアプリで「再読み込み」するか、一度ログアウトしてからログインし直す。
